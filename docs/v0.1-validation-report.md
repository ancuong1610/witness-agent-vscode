## Artifact Paths (Source-Verified)

The paths below are derived directly from the implementation source. Each entry notes the exact
file pattern the code produces.

### After `Witness: Initialize Project`

```
.witness/
├── constitution.md
├── index.md
├── current-state.md
├── commands.md
├── templates/
│   ├── session-template.md
│   ├── context-pressure-template.md
│   ├── subagent-report-template.md
│   ├── adr-template.md
│   ├── handover-template.md
│   └── resume-probe-template.md
├── sessions/.gitkeep
├── telemetry/.gitkeep
├── subagents/.gitkeep
├── decisions/.gitkeep
├── handovers/.gitkeep
└── evaluation/.gitkeep
```

### After `Witness: Start Session`

```
.witness/sessions/<session-id>.md                 (opened in editor)
.witness/telemetry/<session-id>/.gitkeep          (session-scoped telemetry subdirectory created)
```

Active session pointer set in extension state (`.witness/.current-session`).

The session ID format is `YYYY-MM-DD-NNN` where `NNN` is a three-digit sequential counter scoped
to the day. Example: `2026-05-14-001`.

### After `Witness: Record Context Snapshot`

```
.witness/telemetry/<session-id>/context-pressure-NNN.md   (opened in editor)
```

The snapshot file is written into a session-scoped subdirectory inside `telemetry/`. The
subdirectory is created by `Witness: Start Session`. The filename pattern is exactly
`context-pressure-NNN.md` with a three-digit zero-padded ordinal scoped to that session
subdirectory (resets per session).

### After `Witness: Observe Workspace`

```
.witness/sessions/<session-id>-observation-NNN.md   (opened in editor)
```

Three-digit zero-padded ordinal scoped to the session.

### After `Witness: Create ADR`

```
.witness/decisions/ADR-NNNN-<slug>.md   (opened in editor)
```

Four-digit zero-padded project-wide sequential number. Slug is derived from the title using
a slugify function (lowercase, hyphens, max 50 characters).

### After `Witness: Record Subagent Report`

```
.witness/subagents/subagent-NNN.md   (opened in editor)
```

Three-digit zero-padded project-wide sequential number (not session-scoped).

### After `Witness: Assess Continuity Risk`

```
.witness/sessions/<session-id>-risk-NNN.md   (opened in editor)
```

Three-digit zero-padded ordinal scoped to the session. The file includes a five-row risk
dimension table (one row per locked dimension) with auto-computed worst-wins overall level and
the user's confirmed or overridden final level.

### After `Witness: Generate Handover`

```
.witness/handovers/handover-<session-id>-NNN.md   (opened in editor)
.witness/handovers/latest.md                       (simultaneously overwritten)
```

Three-digit zero-padded ordinal scoped to the session. `latest.md` is always overwritten with
the same content as the dated file. The handover ID embedded in the file content is
`handover-<session-id>-NNN` (e.g. `handover-2026-05-14-001-001`).

### After `Witness: Validate Handover`

```
.witness/evaluation/handover-<handover-base>-validation-NNN.md   (opened in editor)
```

The `<handover-base>` segment is the filename of the source handover without its `.md` extension.
When validating `latest.md`, the base is `latest`, producing:
`handover-latest-validation-001.md`. When validating a dated file directly (fallback path), the
base is the dated filename stem, producing:
`handover-handover-<session-id>-NNN-validation-NNN.md`.

The three-digit ordinal is a global counter across all validation reports in `evaluation/`
(not session-scoped). The validator prefers `latest.md` as its source; it falls back to the
highest-ordinal `handover-<session-id>-NNN.md` for the active session if `latest.md` does not
exist.

### After `Witness: Create Resume Probe`

```
.witness/evaluation/resume-probe-<handover-id>-NNN.md   (opened in editor)
```

The `<handover-id>` is extracted from the `## Handover ID` section of the handover file content
(not the filename). If the section is absent or contains only placeholders, the command falls back
to the filename stem. The three-digit ordinal is scoped per handover ID. Example:
`resume-probe-handover-2026-05-14-001-001-001.md`.

### After `Witness: Compress Current State`

```
.witness/sessions/<session-id>-current-state-NNN.md   (archive — opened beside live file)
.witness/current-state.md                              (live file opened for manual trimming)
```

Three-digit zero-padded ordinal scoped to the session.

## Known Limitations (v1)

1. **No OTel instrumentation.** Risk assessments rely entirely on developer-supplied QuickPick
   estimates. Automated corroboration from telemetry data is not yet available. See
   `docs/otel-evaluation-model.md` for the planned v2 model.

2. **Subagent completion report only.** The v1 subagent model captures a completion report but
   not the full five-stage lifecycle (task contract, context packet, evidence summary, orchestrator
   review). See `docs/subagent-ledger-v0.2.md`.

3. **No `@witness` chat participant.** Witness Agent does not integrate into the VS Code chat
   interface. Interaction is entirely through the command palette.

4. **No dashboard UI.** There is no visual panel for current risk level or active session state.

5. **Git dependency for Observe Workspace.** The `Witness: Observe Workspace` command depends on
   the `vscode.git` extension API. In workspaces where git is not initialized or the git extension
   is inactive, git state is reported as unavailable rather than crashing.

6. **Context pressure is estimated, not measured.** The `Witness: Record Context Snapshot` command
   accepts a developer-supplied estimate of context window usage. There is no automated token
   counting or context window API integration in v1.

7. **No marketplace publishing.** The extension runs only in development mode via F5. Marketplace
   publishing infrastructure (publisher ID, VSIX packaging, CI) has not been established.

8. **Detailed artifact evidence not stored.** Concrete artifact samples (filled-in file contents,
   file listings) from smoke validation sessions are not committed to the repository. The artifact
   path specifications in this document are source-verified but not accompanied by example output
   files.
